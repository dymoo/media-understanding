# Three-Tier Media Understanding Plan

## Assumptions

- **Three-tier cost model:**
  1. **Probe** (cheap, batch-safe) — header-only metadata via `probeMedia()`. Safe for 50+ files.
  2. **Heavy analysis** (expensive, single-file) — transcription, grid extraction, frame extraction. One file per MCP call.
  3. **Orchestration** (agent-level, not MCP-level) — multi-file concurrency via subagents, not MCP batching.
- Add a new `probe_media` MCP tool (separate tool, NOT a flag on `understand_media`). Separate tools are the unit of discoverability for LLMs — a `metadata_only` flag buried in an 18-param schema gets missed.
- Remove `file_paths`, `glob`, `max_files`, `batch_concurrency` from `understand_media`. Make `file_path` required.
- Add `fileSizeBytes` to `MediaInfo` (agents need file size for planning decisions).
- Enforce a hard response cap using exact serialized MCP payload characters, default `32000`.
- Prefer higher-quality images over silently over-compressing. Auto-fit under budget when image count is implicit; fail with a teaching error when explicit count is too large.
- Adopt Simon Willison-inspired techniques: multiple separate images per turn, uniform time-distributed sampling, visible timestamps on every frame. Credit in README.
- Be conservative by default. For long or massive files, fail fast unless the caller narrows with explicit `start_sec`/`end_sec`.
- README visuals should use actual library output from explicitly approved local media.
- README MUST include subagent orchestration guidance as a "good use guideline" (not enforcement).

## Open Questions

- None blocking. Direction is clear.

## LLM Workflow Goal

The MCP surface teaches an LLM a three-step workflow via tool names and descriptions alone:

1. **Discover** — `probe_media` with glob/paths. Cheap, batch-safe. Returns type, duration, resolution, codecs, file size.
2. **Analyze** — Pick files that matter, call heavy tools one at a time:
   - `understand_media(file_path)` for single-file full analysis (metadata + transcript + grids)
   - `get_transcript(file_path)` when speech matters most
   - `get_video_grids(file_path)` for visual-first understanding
   - `get_frames(file_path, timestamps)` for exact moments
3. **Iterate** — Use transcript timestamps to target grids. Use grid timestamps to target exact frames. Narrow the window, don't widen it.

For multiple files, the agent orchestrator should launch subagents (context-isolated) each with MCP access and a clear intention. Subagents return distilled results; their context is discarded. This prevents context compression/churn.

## Final MCP Tool Surface (5 tools)

### `probe_media` (NEW — lightweight, batch-safe)

**Purpose:** Discover and triage media files before committing to heavy analysis.

**Schema:**

- `file_path`: string (optional) — single file
- `file_paths`: string[] (optional) — explicit batch
- `glob`: string (optional) — glob pattern
- `max_files`: int (optional, default 50, absolute max 200)

Exactly one of `file_path`, `file_paths`, `glob` required.

**Returns per file:** `MediaInfo` with `fileSizeBytes` added: path, type, duration, width, height, fps, videoCodec, audioCodec, sampleRate, channels, fileSizeBytes.

**Why generous limits:** `probeMedia()` reads headers only (~5-50ms/file). 50 files × ~400 chars = ~20K chars, well within budget. No images, no decoding. The agent needs to scan directories to decide what to analyze.

**No `max_total_chars`:** Probe returns text only. Budget management adds complexity to a tool whose whole point is simplicity.

### `understand_media` (SIMPLIFIED — single-file only)

**Changes from current:**

- Remove: `file_paths`, `glob`, `max_files`, `batch_concurrency`
- `file_path` becomes **required** (was optional)
- All other params unchanged

**Description teaches:** "Use this as your first call for any single media file. For multiple files, use `probe_media` first to discover, then call this once per file."

### `get_video_grids` — unchanged (already single-file)

### `get_frames` — unchanged (already single-file)

### `get_transcript` — unchanged (already single-file)

## Phase 2 — Huge-File Safety And Streaming

**Outcome:** The library behaves predictably on very large media and fails safely before expensive work.

**Work:**

1. Add file-size/duration preflight policy for heavy operations; fail fast with guidance when a file is too large for a sensible single MCP response
2. Prefer explicit windowing (`start_sec`/`end_sec`) for large media instead of hidden auto-chunking
3. Replace unbounded transcript cache with a bounded/LRU-style cache; skip caching oversized results
4. Avoid assembling very large transcript strings before truncation where possible
5. Revisit frame extraction buffering so JS doesn't fully buffer large frame outputs
6. Bound in-process concurrency for frame/grid extraction
7. Add natural-language errors with recovery guidance: narrow `start_sec`/`end_sec`, ask for transcript-only first, request fewer visuals, retry on a shorter window

**Critical Files:**

## Phase 3 — README Rewrite With Subagent Orchestration Guidance

**Outcome:** README teaches both the tool workflow AND the multi-file orchestration pattern. Any LLM (or human) reading it understands the three-tier model immediately.

**Work:**

1. **"How It Works" section** (near top):
   - Three-step: probe → analyze → iterate
   - Visual showing the flow (real library output)

2. **"Recommended LLM Workflow" section:**
   - Step 1: Discover with `probe_media` (cheap, batch)
   - Step 2: Analyze with heavy tools (one file at a time)
   - Step 3: Iterate (transcript timestamps → grids → frames)
   - Per-type recommendations: image → `understand_media`, audio → `get_transcript`, short video → `understand_media`, long video → `get_transcript` or `get_video_grids` first

3. **"Processing Multiple Files" section** (NEW — subagent orchestration):
   - Why not batch heavy operations: context bloat, compression, lost details
   - The subagent pattern:
     - `probe_media` everything first (cheap triage)
     - Launch subagent(s) per file with MCP access
     - Pass **intention** to each subagent: "why are we analyzing this?", "what do we want?"
     - Subagents are **sacrificial** — their context is discarded after returning distilled results
     - Orchestrator synthesizes results without accumulating raw media context
   - When NOT to use subagents: single file, quick metadata checks, 2-3 small files
   - Explicit: this is a **guideline**, not enforcement. The MCP server doesn't prevent serial `understand_media` calls in one context. But analysis quality degrades as context fills up.

4. **Tool descriptions updated** to teach the workflow:
   - `probe_media`: "Use this FIRST. Cheap. Batch-safe."
   - `understand_media`: "Use AFTER probe, on individual files. Heavy. For multiple files, consider subagents."
   - `get_transcript`: "When speech content matters most."
   - `get_video_grids`: "For visual inspection without transcription."
   - `get_frames`: "For exact moments identified from transcript/grid timestamps."

5. Replace synthetic SVG visuals with real library-generated outputs from approved local media
6. Add Simon Willison credit for timestamped multi-image technique
7. Collapse agent setup instructions into `<details>` blocks
8. Update `AGENTS.md` with new tool surface, three-tier model, and subagent pattern
